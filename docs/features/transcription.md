# Audio transcription (`videodoc transcribe`)

## Summary
`videodoc transcribe <project>` is README §17's "Fase 4 — Trascrizione audio": for every video that already has extracted audio (`videodoc extract-audio`), transcribes it with `faster-whisper` into timestamped segments, saves them to `workdir/<id>/transcript/<id>.json` and the project's `project.db` (`transcript_segments` table), and updates that video's `metadata.json`. It is idempotent: if a transcript already exists, the (potentially very expensive, multi-GB-downloading) Whisper engine is never invoked for that video. This is the first command implementing README §11.1's `TranscriptionService`.

## Resolving README's inconsistencies
README describes a transcript segment in three mutually inconsistent shapes: §17's simplified illustration (`{"start": 12.4, "end": 28.7, "text": "..."}`), §9.2's richer example (`start_time`/`end_time` as `"HH:MM:SS"` strings, plus `confidence`), and §31/§30.2's actual SQL schema (`start_seconds`/`end_seconds` as `REAL`). **This implementation treats the SQL schema as authoritative** — `project.db` is README's own stated source of truth (§8.1.1) — so both `transcript_segments` rows and the `transcript/<id>.json` file use `start_seconds`/`end_seconds` (float seconds), never HH:MM:SS strings or the bare `start`/`end` shape. `README.md` §17 has been annotated with a note pointing at this resolution.

Additionally, `transcript_segments.id` is a single global `TEXT PRIMARY KEY` (not composite with `video_id`), so a literal `"seg_0001"` (README §9.2's own example) would collide across videos. Segment ids are generated as `f"{video_id}_seg_{i:04d}"` — one canonical identifier scheme, consistent with how `video_id` (`slugify()`-derived) is already used everywhere else in this codebase, and matching the composite-id style README's own §9.5 knowledge-chunk example uses (`"workshop_01_chunk_0007"`).

## What was implemented
- `core/utils/transcription.py` — mirrors `core/utils/ffprobe.py`/`ffmpeg.py`'s layering exactly: one plain-`Exception` subclass, `TranscriptionError` (deliberately **not** a `VideoDocError` — utils never raise domain exceptions), raised by both:
  - `load_whisper_model(model_name)` — `from faster_whisper import WhisperModel` inside the function (lazy import, purely for import-time cost: `faster-whisper` is a required dependency of this project, not optional, so this is not a "might be missing" guard, though the try/except is kept defensively), then `WhisperModel(model_name)`.
  - `transcribe_audio(model, audio_path, *, language, word_timestamps)` — calls `model.transcribe(...)`, fully consumes the lazy segment generator inside the `try` (faster-whisper can raise mid-iteration), strips segment text, and derives `confidence = exp(avg_logprob)` when available (faster-whisper doesn't expose a direct 0-1 confidence field, only a log-probability, typically ≤ 0 — this is an explicit approximation, not a native model output).
- `core/models/transcript.py::Transcript`/`TranscriptSegment` — mirrors `VideoMetadata`'s exact `load`/`to_json`/`save` triad (`OSError`/`json.JSONDecodeError`/`ValidationError` → new `InvalidTranscriptError`).
- `core/storage/database.py` — `ensure_schema()` now also creates `transcript_segments` (README §31/§30.2 DDL, `IF NOT EXISTS`), so `ingest`'s existing call to `ensure_schema()` provisions this table too, ahead of when `transcribe` first needs it. New `replace_transcript_segments(db_path, video_id, segments)`: a full `DELETE`+`INSERT` per video, not an incremental upsert — a re-transcription is a wholesale regeneration, matching how `sources.yaml` is always fully regenerated on scan, never merged.
- `core/services/transcription_service.py::TranscriptionService.run() -> TranscriptionResult` (`transcribed`/`skipped`/`errors` tuples, no `warnings` bucket — mirrors `AudioExtractionResult`'s shape). Order of checks: `project.db` missing/wrong type → `NoVideosFoundError`/`DatabaseError` (same as `extract-audio`) → zero videos ingested → `NoVideosFoundError` → `config.transcription.engine` validated (only `"faster-whisper"` is implemented, despite README listing 3 candidate tools — an unsupported value raises `TranscriptionEngineError` instead of being silently ignored) → videos partitioned into those with extracted audio (candidates) vs. not (immediate per-video `errors`, not fatal); if **zero** candidates exist, `NoVideosFoundError` pointing at `extract-audio` (same structural-only-when-everything-is-missing split already established).
  - **The Whisper model is loaded once for the whole batch, and only if at least one candidate actually needs fresh transcription** (determined up front by checking which `transcript/<id>.json` files already exist). A fully-skipped rerun never pays the cost of loading the model — which can mean a multi-GB download and real load time — for nothing.
  - **The skip path self-heals.** When a transcript already exists, the engine is never re-invoked, but the on-disk JSON is reloaded and its segments are always rewritten into `transcript_segments` (a cheap `DELETE`+`INSERT`) before `metadata.json` is reconciled. Without this, a transient DB failure right after a successful JSON write in one run would go permanently unnoticed: the next run would see the JSON file, skip everything including the DB write, and the DB would silently stay empty forever. "Skipped" means "the engine wasn't re-invoked," not "nothing else was checked."
  - The transcript JSON is written atomically (`<id>.json.tmp` → `Path.replace()`), the same pattern already applied to the `.wav` file in `extract-audio`, generalized here for consistency (even though the risk window for a JSON write is much smaller than for an `ffmpeg` subprocess).
  - A DB-write failure right after a successful fresh JSON write is not counted as `transcribed` — it's folded into `errors` with a message noting the JSON itself is fine. The next run's skip-path self-heal (above) retries it.
- `videodoc transcribe` CLI: thin wrapper (`ProjectService.load()` → `TranscriptionService(...).run()`), reports counts (`transcribed`/`skipped`) and prints each `errors` entry as a warning — never fails the whole command for a single bad video.

## Design decisions
- `word_timestamps` is passed through to `model.transcribe()` faithfully (a user setting it to `False` genuinely skips that computation in `faster-whisper`), but the resulting per-word data is discarded — README §31 has no table for word-level data, only `transcript_segments`. Documented here explicitly as "not yet consumed," not silently dropped.
- No `device`/`compute_type` field was added to `TranscriptionSection` — not requested by README's spec. **Practical consequence, confirmed during manual verification**: `WhisperModel(model_name)` without an explicit device defaults to auto-detecting hardware, and on a machine where a GPU is detected but its CUDA runtime libraries aren't properly installed (e.g. missing `cublas64_12.dll` on Windows), it fails outright rather than falling back to CPU. Whether that surfaces as a recoverable per-video `TranscriptionError` (if the failure happens inside `transcribe_audio()`, as observed) or as the structural, fatal `TranscriptionEngineError` (if it instead happens inside `load_whisper_model()`'s `WhisperModel(...)` construction) depends on `faster-whisper`/`ctranslate2` internals, not on this codebase — both paths are handled correctly (no crash, no partial state either way), but only the per-video case leaves `exit code` at `0`. Called out in `RUN.md`'s troubleshooting section rather than silently left as a mystery.
- `large-v3` (the current config default) can be slow on CPU and downloads several GB from Hugging Face on first use — documented as a prerequisite callout, not solved with new configuration.
- Only `faster-whisper` is implemented; `whisper.cpp` and "Whisper locale" (README §17's other two listed candidates) are not — an explicit `TranscriptionEngineError` is raised for any other `engine` value rather than silently ignoring it.
- No automatic cleanup of a stale transcript after the source video is reingested with different content — same principle already applied to audio: delete the file manually to force re-transcription.

## Tests
- Unit: `tests/core/test_transcription_utils.py` (`load_whisper_model` success/broken-import/instantiation-failure via `monkeypatch.setitem(sys.modules, "faster_whisper", ...)` — no real model ever loaded; `transcribe_audio` happy path, `avg_logprob is None`, failure during generator iteration), `tests/core/test_transcript.py` (roundtrip, missing/invalid/extra-field), `tests/core/test_database.py` (`transcript_segments` table creation, `replace_transcript_segments` insert/replace/`DatabaseError`).
- Service: `tests/core/test_transcription_service.py` — no `project.db`/empty/directory (mirrors `extract-audio`), zero videos with audio (`NoVideosFoundError`), some-but-not-all missing audio (per-item, not fatal), unsupported engine, model load failure, **model never loaded when everything is already transcribed** (call-count assertion), single video transcribed updates `metadata.json`/JSON/DB rows, skip with call-count zero, **skip self-heals a prior DB failure** (JSON present, DB rows absent, restored without re-invoking the engine), skip-path corrupt-JSON/DB-failure/metadata-failure all reported as errors, one bad video doesn't block others, partial-write leaves no stray `.json`, missing/corrupt `metadata.json` after successful transcription, DB failure after successful JSON write not counted as `transcribed`.
- CLI: `tests/cli/test_cli_transcribe_command.py` (success summary, unknown project, no ingested videos, no extracted audio, unsupported engine, per-video error as non-fatal warning, rerun shows everything skipped).
- Manual: end-to-end run against the real `faster-whisper` package (installed as a required dependency) with the `tiny` model (not the `large-v3` default, to keep the download/runtime reasonable) — confirmed the library integration (`transcribe_audio`'s parsing of the real segment generator) works correctly via a direct CPU-forced diagnostic call, and confirmed the full CLI path handles a genuine real-world failure (a missing CUDA library on this machine) as a per-video warning, `exit code` 0, zero stray files left in `workdir/<id>/transcript/`. **Caveat, found in review**: that graceful handling only holds because the failure happened to surface during `transcribe_audio()` (per-item) in this specific environment/library-version combination; if the same underlying problem instead prevents `load_whisper_model()` from constructing `WhisperModel(...)` at all, it is structural and fatal (`TranscriptionEngineError`, `exit code` 1) by design (see RUN.md §8) — which of the two happens is decided by `faster-whisper`/`ctranslate2` internals, not by this codebase.
- Reproduced on a second real machine (a real GPU detected, CUDA runtime libraries missing) with a real 31-video workshop folder — `ingest`/`extract-audio` completed on all 31 in well under a minute total, confirming the earlier time estimates were conservative. Fixed on that machine by installing `nvidia-cublas-cu12`/`nvidia-cudnn-cu12` via pip **and** adding their `site-packages/nvidia/*/bin` directories to `PATH` for the session — the pip packages alone were not sufficient on Windows (unlike Linux, nothing auto-registers them into the DLL search path). Verified end-to-end afterward: real Italian speech transcribed correctly with plausible confidence scores. Documented as the recommended fix in `RUN.md` §8, not implemented as an automatic step in this codebase (no code touches `PATH` on the user's behalf).
