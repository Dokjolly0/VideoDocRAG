# Changelog

## Step 1 — Project management foundation
- Added project initialization and a validated `config.yaml` schema — see [features/project-initialization.md](features/project-initialization.md)
- Added the local project registry and portable project paths (`--path`, `list`/`link`/`unlink`/`path`) — see [features/project-registry.md](features/project-registry.md)
- Fixed: `init` now registers projects under their canonical slug instead of the raw display name, matching `link`'s existing behavior — see [features/slugify.md](features/slugify.md)
- Fixed: `init` on a path that already holds a *different* project now fails with a clear error instead of silently registering an unintended alias — see [features/slugify.md](features/slugify.md)
- Hardened: a single malformed entry in the local registry can no longer crash `list`/`resolve`, and can always be removed via `unlink` — see [features/project-registry.md](features/project-registry.md)

## Step 2 — Source scanning and external paths
- Added support for external source paths: `paths.videos`/`attachments`/`codebase` can be an absolute path referencing material outside the project, with matching `init --videos`/`--attachments`/`--codebase` options — see [features/external-source-paths.md](features/external-source-paths.md)
- Added `videodoc scan`, which enumerates a project's sources (respecting `config.scan` exclusions) and writes `sources.yaml` — see [features/scan.md](features/scan.md)
- Fixed: `workdir`/`indexes`/`output`/`database` (and relative `videos`/`attachments`/`codebase`) now also reject `..` parent-traversal segments, not just anchored forms — a relative value could otherwise still escape the project folder once joined with it — see [features/external-source-paths.md](features/external-source-paths.md)
- Fixed: `ProjectService.init()` now validates a new project's `config.yaml` (including `--videos`/`--attachments`/`--codebase`) before creating any folder, so an invalid value never leaves partial project state on disk — see [features/external-source-paths.md](features/external-source-paths.md)
- Fixed: added `core/utils/paths.py` as the single shared definition of path classification (`is_external_source_path`, `has_any_anchor`, `has_ambiguous_anchor`, `has_parent_traversal`), used by both config validation and path resolution/reporting (`resolve_source_path`, `SourceScanService._resolve_report`). Previously validation used `PureWindowsPath` while resolution used the platform-dependent `Path` — harmless on Windows (where they agree) but the two could have silently disagreed on a non-Windows host — see [features/external-source-paths.md](features/external-source-paths.md)
- Hardened: `scan`'s directory walker no longer silently drops subdirectories it can't read (e.g. permission denied) — walk errors are now collected into `sources.yaml`'s new `scan_errors` field and surfaced as CLI warnings, never silently producing an incomplete-looking-complete manifest — see [features/scan.md](features/scan.md)
- Fixed: `codebase_is_present()` no longer lets an unreadable-but-existing `codebase/` root (e.g. permission denied) crash `videodoc scan` with an uncaught exception before `scan_codebase()`'s own error handling could even run — it now reports into the same `scan_errors` list — see [features/scan.md](features/scan.md)
- Docs: the stale `sources.yaml` example in README §14 (which showed `config.yaml` content) was replaced with a real `SourceManifest`-shaped example, and the §15.1 `scan` output example now matches the actual CLI output (`Codebase: present (N files)`, `Excluded file patterns`)

## Step 2.1 — Cross-platform support (Linux/macOS)
- Redesigned `core/utils/paths.py` to classify paths using the host's native `pathlib` semantics (`PurePath`, auto-selecting `PureWindowsPath`/`PurePosixPath`) instead of always `PureWindowsPath`, with an injectable `path_class` parameter so both Windows and POSIX rules are verified deterministically from a single dev machine — see [features/external-source-paths.md](features/external-source-paths.md)
- Added `.github/workflows/tests.yml`: the test suite now runs on `windows-latest`/`ubuntu-latest`/`macos-latest` (Python 3.11 and 3.13) on every push/PR — real multi-OS confirmation, not just "should work in theory"
- Added `.gitattributes` (LF line endings in the repository) and extended `.gitignore` with `.DS_Store`/`*.swp`/`*~` — repository hygiene for contributors on Linux/macOS
- Rewrote `RUN.md` with parallel Windows/Linux/macOS sections throughout (setup, venv activation, environment variables, path examples); the Windows-only Microsoft Store Python note stays labeled as such rather than inventing an unverified equivalent for other OSes
- Docs: README §8.1.2 registry path table corrected (was conflating Linux and macOS default data directories, and used the wrong case for the app-data folder name — `platformdirs` actually uses lowercase `videodoc`); added a note on `videos`/`attachments`/`codebase` absolute-path portability (§8.2)

## Step 2.2 — Exclude build/IDE output by default, for every supported language
- Fixed: `DEFAULT_EXCLUDES` (README §8.3, `core/storage/filesystem.py`) now also excludes `bin/`, `obj/`, `.vs/` — found via a manual `scan` test against a real multi-project .NET/Vue codebase, where build output (`obj/**/*.json`) and Visual Studio IDE state were being indexed as if they were source, since only `node_modules/`-style JS/Python exclusions existed before — see [features/external-source-paths.md](features/external-source-paths.md)
- Extended: `DEFAULT_EXCLUDES` grew from 26 to 35 entries to cover build output and IDE state for languages beyond JS/Python/.NET — `.tox/` (Python), `target/`/`.gradle/` (JVM: Java/Kotlin/Scala), `vendor/` (Go/PHP, vendored third-party dependencies), `.dart_tool/` (Dart/Flutter), `Pods/`/`DerivedData/` (iOS/Swift), and IDE state `.idea/` (JetBrains), `.vscode/` (VS Code), `.settings/` (Eclipse). Deliberately does **not** add `packages/`: it is a legacy NuGet artifact folder, but also the standard *source* directory name in JS/TS monorepos (Yarn/npm/pnpm workspaces, Lerna, Turborepo, Nx) — excluding it by default would risk silently hiding real code, unlike every other entry in this list — see [features/scan.md](features/scan.md)

## Step 3 — Video ingestion
- Added `videodoc ingest`: for every video in `videos/` (internal or external), hashes the file, probes duration/format/resolution/codec via `ffprobe`, registers it in the project's own `project.db` (README §31 schema), and creates `workdir/<id>/{audio,frames,transcript,ocr,chunks}/` + `metadata.json` — see [features/ingest.md](features/ingest.md)
- `ingest` is the first command allowed to require an external tool (`ffprobe`, part of FFmpeg) — `init`/`scan`/`list`/`link`/`unlink`/`path` remain dependency-free (README §37.8)
- Idempotent by content hash: an unchanged video is skipped without even being reprobed; a changed video is reprocessed and a warning is raised about possibly-stale sibling artifacts in its `workdir/` (never deleted automatically, same "never delete without being asked" precedent as `unlink`)
- Zero video files found is a hard failure for `ingest` (`NoVideosFoundError`), unlike `scan`, which treats it as a normal, non-failing report — README §15.1 already framed this as ingestion's responsibility, not scan's
- Hardened: a video id collision (two different files whose names both slugify to the same identifier) is detected both within a single run and against a previous run's `project.db`, raising `VideoIdCollisionError` instead of silently overwriting an unrelated video's data
- Fixed (caught by the test suite's `ResourceWarning`, not by inspection): `sqlite3.Connection` used bare as a context manager only wraps commit/rollback, it does not close the connection — `core/storage/database.py` now chains `contextlib.closing(...)` to actually release the file handle on every query
- Added `docs/commands.md`: a single always-up-to-date reference listing every implemented CLI command with its syntax and expected output

## Step 4 — Audio extraction
- Added `videodoc extract-audio`: for every video already registered by `ingest`, extracts mono 16kHz PCM WAV audio via FFmpeg into `workdir/<id>/audio/<id>.wav` and updates that video's `metadata.json` (`audio_path`) — see [features/audio-extraction.md](features/audio-extraction.md)
- Added `core/storage/database.py::list_videos()`, the first batch-read over every registered video (previously only single-video lookup via `get_video` existed)
- Hardened: audio is written to a temporary `.tmp` path and moved onto the final `.wav` only via an atomic `Path.replace()` after FFmpeg succeeds — an interrupted FFmpeg process (crash, disk full, Ctrl+C) can no longer leave a partial/corrupt file at the path the idempotency check reads, which would otherwise be misread as "already extracted" forever with no way to detect or fix it
- Idempotent by file presence: if a video's `.wav` already exists, FFmpeg is never re-invoked; `metadata.json` is still reconciled if it was left holding `ingest`'s original folder-only placeholder
- Fixed: `db_path.exists()` is checked before any `sqlite3.connect()` call, so a project where `ingest` was never run raises `NoVideosFoundError` without silently creating an empty `project.db` file as a side effect of the failure path itself
- README §12.1/§16 updated to reflect the dedicated `extract-audio` command, which the README's own command list previously omitted (it jumped straight from `ingest` to the not-yet-implemented `transcribe`)

## Step 5 — Audio transcription
- Added `videodoc transcribe`: for every video with extracted audio, transcribes it via `faster-whisper` into `workdir/<id>/transcript/<id>.json` and the new `transcript_segments` table in `project.db`, and updates that video's `metadata.json` (`transcript_path`) — see [features/transcription.md](features/transcription.md)
- Added `faster-whisper` as a required dependency (`pyproject.toml`); the actual `import faster_whisper` stays lazy (inside a function, not module-level) purely to avoid its import-time cost for every command that doesn't transcribe anything
- Resolved a 3-way inconsistency in README's own description of a transcript segment (§17's bare `start`/`end`, §9.2's `HH:MM:SS` strings, §31/§30.2's `start_seconds`/`end_seconds` REAL) by treating the SQL schema as authoritative for both the DB rows and the JSON file — see features doc for the full rationale
- Hardened: the (potentially multi-GB-downloading) Whisper model is loaded once per run, and only if at least one video actually needs fresh transcription — a fully-skipped rerun never pays that cost
- Hardened: the skip-because-already-transcribed path still rewrites the video's `transcript_segments` rows from the on-disk JSON every run (cheap), so a transient database failure right after a successful JSON write in an earlier run self-heals on the next run instead of silently staying empty forever
- Idempotent by file presence, same pattern as `extract-audio`: if a video's transcript JSON already exists, the engine is never re-invoked; atomic `.tmp` + `Path.replace()` write, same crash-safety rationale as the `.wav` file
- README §17 annotated with a note documenting the schema resolution above and that only `faster-whisper` (of the three tools listed) is currently implemented
- Added `docs/commands.md` entry and `RUN.md` §5.9, including a troubleshooting note for a real failure encountered during manual verification: `WhisperModel` defaults to auto-detecting hardware and can fail outright (not fall back to CPU) on a machine with a partially-detected GPU missing its CUDA runtime libraries -- handled gracefully as a per-video warning, not a crash
