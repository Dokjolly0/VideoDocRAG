# Changelog

## Unreleased — Intelligent chunking
- Added `videodoc chunk`: for every video with transcript/OCR/code inputs, builds deterministic time chunks in `workdir/<id>/chunks/<id>.json`, replaces that video's rows in the new `chunks` table, and updates `metadata.json` (`chunks_path`) — see [features/chunking.md](features/chunking.md).
- Added transcript-backed windowing with config-driven duration bounds (`chunking.min_duration_seconds` / `max_duration_seconds`) plus pause splitting, OCR/code enrichment inside each time interval, and OCR/code-only fallback windows when transcript is not available.
- Added one code-specific chunk per `code_blocks` row (`source_type="code"`) so the future embedding/indexing phase can index code as a standalone retrieval document while still keeping it attached to surrounding narrative chunks.
- Added `core/models/chunk_manifest.py` with transcript/frame/code input signatures, ensuring reruns of transcription, OCR, code detection, or frame extraction trigger a fresh chunking pass instead of reusing stale chunks.
- Added `replace_chunks()`/`list_chunks()` and `ChunkRow` in `core/storage/database.py`; chunking reads upstream tables only and writes only the `chunks` table plus `metadata.json`.
- Added tests for manifest loading, DB helpers, chunking service idempotency/self-heal/reprocessing/fallback behavior, and CLI wiring.
- Docs: README §21, `docs/commands.md`, `RUN.md`, and `docs/features/chunking.md` now document `videodoc chunk`.

## Unreleased — Code extraction from OCR
- Added `videodoc code`: for every video with OCR already present, classifies OCR text, extracts code-like blocks, deduplicates repeated blocks across frames, writes `workdir/<id>/code/<id>.json`, replaces the video's rows in `project.db`'s new `code_blocks` table, updates only `frames.contains_code`, and writes `workdir/<id>/code/code_review_report.md` for blocks that need human review — see [features/code-extraction.md](features/code-extraction.md).
- Added deterministic OCR-code classification and validation in `core/utils/code_detection.py`: parser-backed validation for JSON/YAML/Python; simple rule verification for terminal commands, file paths, and error messages; conservative classification for JavaScript/TypeScript/HTML/CSS/SQL/Dockerfile, marked for review in strict mode until parser-specific validation exists.
- Added `core/models/code_manifest.py` with full input-frame signatures (`frame_id`, timestamp, perceptual hash, OCR text hash, OCR confidence), so a `videodoc frames` or `videodoc ocr` rerun cannot silently reuse code blocks extracted from stale OCR.
- Added `replace_code_blocks()`/`list_code_blocks()` and `replace_frame_code_flags()` to `core/storage/database.py`; the latter rewrites only `contains_code` and deliberately leaves `ocr_text`/`ocr_confidence`/`perceptual_hash` untouched.
- Added tests for classification, manifest roundtrip/validation, database helpers, `CodeService` idempotency/self-heal/reprocessing/review-report behavior, and CLI wiring through a stubbed `init -> ingest -> frames -> ocr -> code` flow.
- Docs: README §20, `docs/commands.md`, `RUN.md`, and OCR/frame feature notes now describe `videodoc code` as the implemented owner of `contains_code`.

## Unreleased — Transcription throughput optimization
- Replaced PySceneDetect in `videodoc frames` with FFmpeg scene detection (`select=gt(scene\,threshold)`), removing the Python scene-detection dependency and keeping scene failures as per-video warnings.
- Added `frames.scene_threshold` / `--scene-threshold` as a semantic frame-selection setting stored in `frames.json`; legacy manifests without it are re-extracted once when scene detection is enabled.
- Added `frames.hwaccel` / `--hwaccel auto|cuda|none` for FFmpeg decode acceleration. CUDA is performance-only, not part of idempotency, and failed CUDA passes retry once on CPU.
- Added conservative FFmpeg CUDA probing, bounded GPU decode concurrency, and parallel perceptual-hash precomputation for larger frame batches.
- Optimized `videodoc transcribe` for long CUDA workloads: `auto` now prefers batched faster-whisper inference on CUDA, one video worker, VAD, greedy decoding (`beam_size: 1`), and `word_timestamps: false` because word-level data is not persisted.
- Added GPU-aware CUDA planning from dedicated VRAM only: NVML via `nvidia-ml-py` with `nvidia-smi` fallback, dynamic `compute_type`/`batch_size`, and no use of Windows shared GPU memory for sizing.
- Added CUDA OOM recovery: compute-type downgrade on model-load OOM, serial first-video pre-flight with batch halving/reload, and one half-batch retry for later videos.
- Added transcription runtime config fields and CLI overrides: `device`, `compute_type`, `mode`, `workers`, `cpu_threads`, `batch_size`, `beam_size`, `best_of`, `vad_filter`, `chunk_length_seconds`, `condition_on_previous_text`, plus `--word-timestamps/--no-word-timestamps`.
- Enriched `videodoc doctor` with detected GPU name, dedicated VRAM, compute capability, driver, probe source, and the resulting CUDA auto plan.
- Added tests for GPU probing, hardware planning, batched pipeline wrapping, service-level CUDA auto/OOM runtime paths, doctor GPU reporting, and CLI propagation.

## Step 3 — Video ingestion
- Added `videodoc ingest`: for every video in `videos/` (internal or external), hashes the file, probes duration/format/resolution/codec via `ffprobe`, registers it in the project's own `project.db` (README §31 schema), and creates `workdir/<id>/{audio,frames,transcript,ocr,chunks}/` + `metadata.json` — see [features/ingest.md](features/ingest.md)
- `ingest` is the first command allowed to require an external tool (`ffprobe`, part of FFmpeg) — `init`/`scan`/`list`/`link`/`unlink`/`path` remain dependency-free (README §37.8)
- Idempotent by content hash: an unchanged video is skipped without even being reprobed; a changed video is reprocessed and a warning is raised about possibly-stale sibling artifacts in its `workdir/` (never deleted automatically, same "never delete without being asked" precedent as `unlink`)
- Zero video files found is a hard failure for `ingest` (`NoVideosFoundError`), unlike `scan`, which treats it as a normal, non-failing report — README §15.1 already framed this as ingestion's responsibility, not scan's
- Hardened: a video id collision (two different files whose names both slugify to the same identifier) is detected both within a single run and against a previous run's `project.db`, raising `VideoIdCollisionError` instead of silently overwriting an unrelated video's data
- Fixed (caught by the test suite's `ResourceWarning`, not by inspection): `sqlite3.Connection` used bare as a context manager only wraps commit/rollback, it does not close the connection — `core/storage/database.py` now chains `contextlib.closing(...)` to actually release the file handle on every query
- Added `docs/commands.md`: a single always-up-to-date reference listing every implemented CLI command with its syntax and expected output

## Step 4 — Audio extraction
- Added `videodoc extract-audio`: for every video already registered by `ingest`, extracts mono 16kHz PCM WAV audio via FFmpeg into `workdir/<id>/audio/<id>.wav` and updates that video's `metadata.json` (`audio_path`) — see [features/audio-extraction.md](features/audio-extraction.md)
- Added `core/storage/database.py::list_videos()`, the first batch-read over every registered video (previously only single-video lookup via `get_video` existed)
- Hardened: audio is written to a temporary `.tmp` path and moved onto the final `.wav` only via an atomic `Path.replace()` after FFmpeg succeeds — an interrupted FFmpeg process (crash, disk full, Ctrl+C) can no longer leave a partial/corrupt file at the path the idempotency check reads, which would otherwise be misread as "already extracted" forever with no way to detect or fix it
- Idempotent by file presence: if a video's `.wav` already exists, FFmpeg is never re-invoked; `metadata.json` is still reconciled if it was left holding `ingest`'s original folder-only placeholder
- Fixed: `db_path.exists()` is checked before any `sqlite3.connect()` call, so a project where `ingest` was never run raises `NoVideosFoundError` without silently creating an empty `project.db` file as a side effect of the failure path itself
- README §12.1/§16 updated to reflect the dedicated `extract-audio` command, which the README's own command list previously omitted (it jumped straight from `ingest` to the not-yet-implemented `transcribe`)

## Step 5 — Audio transcription
- Added `videodoc transcribe`: for every video with extracted audio, transcribes it via `faster-whisper` into `workdir/<id>/transcript/<id>.json` and the new `transcript_segments` table in `project.db`, and updates that video's `metadata.json` (`transcript_path`) — see [features/transcription.md](features/transcription.md)
- Added `faster-whisper` as a required dependency (`pyproject.toml`); the actual `import faster_whisper` stays lazy (inside a function, not module-level) purely to avoid its import-time cost for every command that doesn't transcribe anything
- Resolved a 3-way inconsistency in README's own description of a transcript segment (§17's bare `start`/`end`, §9.2's `HH:MM:SS` strings, §31/§30.2's `start_seconds`/`end_seconds` REAL) by treating the SQL schema as authoritative for both the DB rows and the JSON file — see features doc for the full rationale
- Hardened: the (potentially multi-GB-downloading) Whisper model is loaded once per run, and only if at least one video actually needs fresh transcription — a fully-skipped rerun never pays that cost
- Hardened: the skip-because-already-transcribed path still rewrites the video's `transcript_segments` rows from the on-disk JSON every run (cheap), so a transient database failure right after a successful JSON write in an earlier run self-heals on the next run instead of silently staying empty forever
- Idempotent by file presence, same pattern as `extract-audio`: if a video's transcript JSON already exists, the engine is never re-invoked; atomic `.tmp` + `Path.replace()` write, same crash-safety rationale as the `.wav` file
- README §17 annotated with a note documenting the schema resolution above and that only `faster-whisper` (of the three tools listed) is currently implemented
- Added `docs/commands.md` entry and `RUN.md` §5.9, including a troubleshooting note for a real failure encountered during manual verification: `WhisperModel` defaults to auto-detecting hardware and can fail outright (not fall back to CPU) on a machine with a partially-detected GPU missing its CUDA runtime libraries -- handled gracefully as a per-video warning, not a crash

## Step 6 — Machine health check and guided fixes (doctor, setup)
- Added `videodoc doctor`: the first machine-scoped command (no project argument, no `ProjectService.load()`) -- checks Python version, FFmpeg (`ffprobe`+`ffmpeg` as one check, not two), faster-whisper importability, GPU/CUDA runtime loadability (the exact real bug hit twice during Step 5 manual testing), the local project registry, and the default projects home directory -- see [features/doctor-setup.md](features/doctor-setup.md)
- Added `videodoc setup`: applies doctor's findings -- pip-installable fixes (venv-scoped, reversible) run automatically without a prompt; system-level fixes (installing FFmpeg via `winget`/`apt`/`brew`) require explicit per-action confirmation; a manual-only fix (e.g. the Windows PATH step for cuBLAS) is never auto-applied and is printed instead
- Added a two-tier CUDA check (`core/utils/cuda.py`): `ctranslate2.get_cuda_device_count()` alone is not sufficient -- verified empirically that it reports a nonzero count even on a machine where cuBLAS itself can't load -- so a direct `ctypes.CDLL(..., winmode=0)` probe of the cuBLAS library is also required
- Fixed (found only through a real manual run, not by inspection): `ctypes.CDLL()`'s default Windows behavior since Python 3.8 no longer searches `PATH` at all (an anti-DLL-hijacking hardening) -- the CUDA probe was reporting a false "not loadable" even with the exact PATH configuration that lets the real `faster-whisper` code path succeed; fixed by passing `winmode=0` to restore legacy PATH-inclusive search
- Added `ProjectRegistry.last_load_was_corrupted` and `.registry_path`, minimal read-only additions so `doctor` can report registry health without reimplementing `registry.json` parsing or reaching into a private attribute
- No new `VideoDocError` subclass: neither command ever raises past its own boundary -- `doctor` always returns a full report, `setup` always reports what it did/skipped, matching the per-item (not domain-exception) failure pattern already used by `extract-audio`/`transcribe`
- The interactive confirmation for system-level fixes lives entirely in `cli/commands/setup.py`, never in `core/` -- keeps `core/` usable from a future GUI (a modal dialog, not a terminal prompt) and matches README §12.4's CLI/core split

## Step 7 — Consistent, well-formatted CLI output
- Added `render_summary_table()` (`cli/output.py`): every multi-field summary command (`scan`, `ingest`, `extract-audio`, `transcribe`) now renders its counts as a Rich table (header-less, bold field column, folding value column) instead of a single colon-joined text line -- matching the visual quality already established by `videodoc list`'s own table
- Added `print_check_result()` (`cli/output.py`), replacing the old `print_check_error`: `doctor`/`setup` report each check as a colored ASCII status word (`OK`/`WARN`/`ERROR`) padded to a fixed width, aligned left of the check's message -- deliberately still line-based, not a table, since fix messages can be long multi-clause PATH instructions that a table column would truncate or wrap badly
- Found and worked around empirically (verified directly on this machine, not by inspection): raw Unicode status glyphs (checkmark/warning-triangle/cross symbols) crash with `UnicodeEncodeError` on a real legacy-Windows PowerShell console, reproduced with both bare `print()` and Rich's own `Console.print()` -- Rich's legacy-Windows box-drawing substitution table only covers its own table-drawing characters, not arbitrary Unicode symbols. `print_check_result` uses plain ASCII words with Rich color markup instead, confirmed to render correctly on the same broken console with no crash
- `init`/`link`/`unlink` stay single-line confirmations (no tabular data to show) and `path` stays completely untouched (deliberately plain, script-friendly output, e.g. `cd (videodoc path <name>)`) -- scoped out of this step with the user upfront
- Added `tests/cli/test_output.py`, dedicated coverage for both new output helpers, including a regression test asserting `print_check_result` never emits a non-ASCII character for any status
- Docs: `docs/commands.md`, `RUN.md`, every `docs/features/*.md` `## CLI` section, and README §15's `scan` "Output atteso" block all updated with real-captured table/status-word output
- Fixed (found through real manual usage on a machine with the cuBLAS PATH problem): the GPU/CUDA check's PATH fix instructions (`doctor_service.py::_check_cuda`) used a generic `<venv>\...` placeholder instead of a copy-pasteable command -- now resolves the running interpreter's actual `site-packages` directory via `sysconfig.get_paths()["purelib"]` and prints the literal `$env:PATH = "..."` command for that machine's real venv location

## Step 8 — Frame and screenshot extraction
- Added `videodoc frames`: for every ingested video, selects timestamps combining a fixed interval, FFmpeg scene-filter scene changes, and transcript-keyword hits (README §18.3), extracts them via a single FFmpeg `select`+`showinfo` call per video into `workdir/<id>/frames/frame_NNNN.jpg`, and records them in `workdir/<id>/frames/frames.json` and the new `frames` table in `project.db`, updating `metadata.json` (`frames_path`) -- see [features/frame-extraction.md](features/frame-extraction.md)
- Added `Pillow` as a required dependency for perceptual hashing; scene detection uses FFmpeg's scene filter
- Added a perceptual-hash adjacent-frame dedup pass (`core/utils/frame_hash.py`): a boosted (scene/keyword) frame near-identical to the immediately preceding kept frame is dropped; interval frames are never dropped, since they are the guaranteed pacing baseline
- Fixed (found via a real end-to-end run against a video with a hard color cut, not by inspection): plain average-hash (aHash) sets every bit relative to the image's own mean, so two *different* flat-colored frames (e.g. a solid red frame and a solid blue frame -- routine in screen recordings: blank terminals, solid-color slides) hash identically on spatial bits alone; a quantized mean-brightness suffix was added to `average_hash`, compared *numerically* (not by bit-XOR, which does not correlate with closeness for a magnitude value) via the new `is_near_duplicate()`, so genuinely different flat colors are no longer misclassified as duplicates
- `frames.json` (mirrors `transcript.json`'s role for `transcribe`) makes the phase idempotent by presence and self-healing: a prior transient database failure is repaired from the on-disk manifest on the next run without re-invoking FFmpeg
- Added `frames.workers`, `frames.scene_threshold`, and `frames.hwaccel` config fields; the keyword list, min candidate gap, and hash-dedup thresholds remain module constants
- Added `docs/commands.md` entry and `RUN.md` §5.10 (doctor/setup renumbered to §5.11/§5.12)
- Verified end-to-end against real `ffmpeg` with synthetic test videos, including the hard-cut clip that surfaced the flat-image hashing bug above
- Fixed (found by code review, then confirmed against a real multi-video project): `videodoc frames` required `ffmpeg` unconditionally at the top of every run, even when every video's `frames.json` already existed and needed only a DB/metadata self-heal -- this broke the self-heal path's own documented guarantee on a machine missing either tool. Availability is now checked only if a pre-scan finds at least one video that actually needs fresh extraction
- Fixed (same review, same real-project confirmation -- the exact scenario hit while testing: a fast `--interval-seconds 30 --no-scene-detection` run followed by a real default-settings run): idempotency was keyed only on `frames.json`'s *presence*, so a rerun with different `--interval-seconds`/`--scene-detection`/`--keyword-boost` silently skipped every video instead of re-extracting with the new settings. `frames.json` now also stores the effective settings it was produced with (`None` for a manifest predating this field, always treated as stale); a mismatch triggers full re-extraction and cleans up any `frame_NNNN.jpg` left over from a denser previous run
- Fixed (found by code review): `core/utils/scene_detection.py` no longer imports the previous Python scene detector; FFmpeg scene detection now uses the same per-video subprocess error boundary as frame extraction
- Fixed (found by code review): an `extract_frames` call that succeeds (matching frame/pts counts) but yields zero frames after matching+dedup was silently written as an empty `frames.json` and reported "Extracted" -- now treated as a per-video error instead
- Removed the Python scene-detection dependency; FFmpeg is the single external tool for both scene detection and frame extraction.

## Step 9 — OCR of extracted screenshots
- Added `videodoc ocr`: for every video with frames already extracted, runs RapidOCR on each frame image and records per-frame text/confidence in `workdir/<id>/ocr/<id>.json` and the `frames` table's `ocr_text`/`ocr_confidence` columns, never touching `perceptual_hash`/`contains_code` (the latter reserved for the not-yet-implemented code-detection phase, README §20) — see [features/ocr.md](features/ocr.md)
- Chosen RapidOCR (`rapidocr` + `onnxruntime`) over README's original PaddleOCR default, following this project's established "Nota implementativa" pattern (see §17/§18): `paddlepaddle`'s Windows install story is notoriously painful (separate CPU/GPU/CUDA-version wheels), RapidOCR is plain pip-installable with no system binary. `config.ocr.engine`'s default is corrected from `"paddleocr"` to `"rapidocr"`
- Verified directly against the installed `rapidocr` 3.9.1 (not assumed from documentation alone): `result.txts`/`result.scores` are both `None` — not empty tuples — when zero text is detected; `run_ocr()` treats this as "ran cleanly, found nothing", not a failure
- Idempotent by `ocr.json`'s stored settings (`engine`/`languages`/`min_confidence`) AND by the current frame-id set: a `videodoc frames` re-run producing a different frame set now also triggers re-OCR, not just an OCR-settings change — the OCR-phase-specific idempotency edge, absent from `frames.json`'s own settings-only comparison
- Added `ocr.workers` config field (mirrors `frames.workers`/`audio.workers`); one RapidOCR engine instance is loaded per video (never shared across the `ThreadPoolExecutor`'s worker threads, since RapidOCR's thread-safety for concurrent inference is unverified), and frames within one video are OCR'd sequentially, not in parallel
- Added `core/storage/database.py::list_frames()`/`update_frame_ocr()` — a partial per-row `UPDATE` of only `ocr_text`/`ocr_confidence`, deliberately not reusing `replace_frames` (which would require reconstructing/risk clobbering `perceptual_hash` and `contains_code`)
- `min_confidence` is a filter on text, not a validity flag: a below-threshold frame is still recorded (`ocr_text=""`, real confidence kept), distinguishing "OCR ran and found low-confidence noise" from "OCR never ran on this frame" (stays `NULL`, automatically retried on the next run since a failed/skipped frame's id is absent from the manifest)
- Added `rapidocr>=3.0,<4.0` and `onnxruntime>=1.15` as required dependencies (not `rapidocr-onnxruntime`, an effectively abandoned predecessor package with no release in ~18 months)
- No `doctor` check added for RapidOCR: availability is checked lazily inside `OCRService` only when at least one video actually needs fresh OCR
- Investigated wiring `config.ocr.languages` to RapidOCR's per-language recognition models (`Rec.lang_type`/`Rec.ocr_version`/`Rec.model_type`, which require real `Enum` values, not strings, and a one-time model download from modelscope.cn) but verified the default "ch" model already recognizes Italian correctly (~0.99 confidence) on realistic screen-quality text — an earlier "garbled" result was traced to a low-quality test font, not a real model limitation — so `languages` stays recorded-for-idempotency-only rather than adding that fragility for no measured benefit
- Added `docs/commands.md` entry and `RUN.md` §5.11 (doctor/setup renumbered to §5.12/§5.13)
- Verified end-to-end against the real (non-stubbed) `rapidocr` package: a synthetic FFmpeg `drawtext` video run through `init` → `ingest` → `frames` → `ocr`, correctly recognizing burned-in terminal text at ~0.986 confidence, confirming idempotent rerun (engine not reloaded) and that `contains_code` stays untouched
- Fixed (found by code review, confirmed end-to-end): a plain idempotent `videodoc frames` rerun (settings unchanged, self-heal path only) rebuilt `frames` DB rows from `frames.json` alone, which never carries `ocr_text`/`ocr_confidence`/`contains_code` — silently wiping any OCR results (or future code-detection flags) already written for those exact frame ids. `FrameExtractionService._process_existing` now reads the current DB values for those three columns first and carries them over for any frame id that already had them
- Fixed (found by code review): `OCRService` recorded `config.ocr.engine` in the manifest but never validated it, so a project whose `config.yaml` still had the pre-correction `paddleocr` default would silently run RapidOCR anyway and write a manifest that misreported which engine actually produced its results — now raises `OCREngineNotSupportedError` unconditionally at the start of every run if `ocr.engine` isn't `"rapidocr"`
- Fixed (found by code review): `rapidocr_available()` only checked for `rapidocr` itself; `rapidocr` does not declare `onnxruntime` as a hard dependency of its own, so a machine with `rapidocr` installed but `onnxruntime` missing passed the up-front check and only failed once every single video's `load_engine()` call was individually attempted (a structural, run-wide problem misreported as N separate per-video warnings). `rapidocr_available()` now also checks `onnxruntime`
- Fixed (found by code review): a video whose `frames`/`frames.json` had real entries on disk but whose `frames` DB table had zero rows (a DB/table desync, e.g. after a `project.db` rebuild) was silently treated as `skipped`, identical to "videodoc frames was never run" — `OCRService` now distinguishes the two and reports a clear per-video error telling the user to rerun `videodoc frames` to rebuild the database
- Fixed (found by a follow-up code review): the same zero-DB-rows detection above silently swallowed a `frames.json` that exists but fails to parse and fell through to the plain skip path instead of reporting it — now also reported as a clear per-video error
- Fixed (found by a follow-up code review): OCR idempotency compared only the *set* of frame ids, not their content — since frame ids are assigned densely by position (`demo_frame_0001`, `demo_frame_0002`, ...), a `videodoc frames` re-run with different settings that happens to produce the same frame *count* reused the same id set over completely different timestamps/images, so OCR silently skipped and reapplied stale text to the new frames. `OCRManifestEntry` now also stores each frame's `timestamp_seconds`/`perceptual_hash` at OCR time, and idempotency compares the full `(id, timestamp_seconds, perceptual_hash)` signature
